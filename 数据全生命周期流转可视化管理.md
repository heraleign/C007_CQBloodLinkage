# 数据全生命周期流转可视化管理 — 详细设计

---

## 一、文档说明

### 1.1 编制目的

基于客户提供的《数据全生命周期流转可视化管理》技术规范书，结合天源迪科、思特奇、数派科技三家厂商的数据汇聚接口调研结果，输出完整的详细设计方案，指导开发团队实施落地。

### 1.2 术语定义

| 术语 | 说明 |
|------|------|
| 血缘（Lineage） | 数据从产生到消费的完整流转路径 |
| 入湖 | 外部业务系统数据接入数据湖 |
| 湖内加工 | 数据湖内部的ETL加工处理 |
| 出湖 | 数据从数据湖输出到外部系统/服务 |
| TDP | 大数据调度平台 |
| AST | 抽象语法树（Abstract Syntax Tree） |
| 血缘节点 | 血缘链路中的一个数据实体（表、文件、Topic等） |
| 血缘边 | 两个血缘节点之间的数据流转关系 |

| 《数据全生命周期流转可视化管理》 | 客户技术规范书，定义10大能力项 |

## 三、统一血缘存储模型设计

### 3.1 血缘节点模型

血缘节点是血缘链路中的基本单元，需统一抽象不同类型的数据实体。

#### 3.1.1 节点类型枚举

| 节点类型编码 | 节点类型名称 | 说明 | 唯一标识构成 |
|-------------|-------------|------|-------------|
| EXT_DB | 湖外数据库表 | 外部业务系统数据库表 | serverIP:port/instance/schema/tableName |
| EXT_FILE | 湖外文件服务器 | 外部文件数据源 | protocol://serverIP:port/filePath/fileNameFormat |
| EXT_KAFKA | 湖外Kafka Topic | Kafka消息队列 | clusterServer/topicName |
| LAKE_TABLE | 湖内大数据平台表 | Hive/Spark表 | clusterID/database/tableName |
| OUT_FILE | 出湖文件（行云） | 输出到行云文件服务器 | fileServerID/filePath |
| OUT_FILE_EXT | 出湖文件（非行云） | 输出到非行云文件服务器 | serverIP:port/filePath |
| OUT_DB | 出湖数据库 | 输出到中心内系统数据库 | serverIP:port/instance/schema/tableName |
| OUT_ES | 出湖ES | 发布到ES提供API查询 | esCluster/indexName |
| OUT_API | 出湖API | 对外API服务 | apiID/apiPath |

## 四、能力一：端到端全链路血缘贯通

### 4.1 入湖血缘采集（对接数据汇聚系统）

#### 4.1.1 设计概述

通过定时任务（凌晨1:00）调用数据汇聚系统提供的4个接口，获取入湖采集配置信息，自动生成入湖阶段的血缘节点和边。

#### 4.1.2 接口对接详细设计

**4.1.2.1 认证模块**

```java
public class AuthTokenGenerator {
  
    private static final String SYSTEM_CODE = "dxmh";
  
    /**
     * 生成认证Token
     * 规则：将{code, timestamp}拼接为JSON后AES加密
     */
    public static String generateAuthToken(String aesKey) {
        long timestamp = System.currentTimeMillis();
        String json = String.format(
            "{\"code\":\"%s\",\"timestamp\":\"%d\"}", 
            SYSTEM_CODE, timestamp
        );
        return AESUtil.encrypt(json, aesKey);
    }
}
```

**4.1.2.2 汇聚接口适配器**

| 接口 | URL（待分配） | 调用频率 | 数据处理 |
|------|-------------|---------|---------|
| 文件采集方式 | /api/lineage/file-collect | 每日凌晨1:00 | 生成EXT_FILE→LAKE_TABLE血缘 |
| 数据库采集方式 | /api/lineage/db-collect | 每日凌晨1:00 | 生成EXT_DB→LAKE_TABLE血缘 |
| Kafka采集方式 | /api/lineage/kafka-collect | 每日凌晨1:00 | 生成EXT_KAFKA→LAKE_TABLE血缘 |
| 表结构清单 | /api/lineage/table-structure | 每日凌晨1:00 | 更新LAKE_TABLE节点元数据 |

**4.1.2.3 文件采集血缘生成逻辑**

```
输入：文件采集接口返回的data列表
处理流程：
  FOR EACH record IN data:
    1. 构造源节点（EXT_FILE）
       - node_unique_key = record.protocol + "://" + record.serverIP + ":" + record.serverPort + record.filePath + "/" + record.fileNameFormat
       - system_name = record.systemName
       - table_name = record.tableName
     
    2. 构造目标节点（LAKE_TABLE）
       - node_unique_key = defaultCluster + "/" + defaultDB + "/" + record.tableName
       - 从表结构清单接口补充tableCnName等
     
    3. 根据dataType处理：
       - I（新增）: 创建节点和边
       - U（更新）: 更新节点和边
       - D（删除）: 逻辑删除节点和边
     
    4. 构造血缘边
       - lineage_stage = "INGEST"
       - collect_method = "FILE"
       - collect_config_id = record.id
     
    5. 执行血缘标准校验（见4.2节）
    6. 保存（见4.3节）
```

**4.1.2.4 数据库采集血缘生成逻辑**

```
输入：数据库采集接口返回的data列表
处理流程：
  FOR EACH record IN data:
    1. 构造源节点（EXT_DB）
       - node_unique_key = record.serverIP + ":" + record.serverPort + "/" + record.instanceName + "/" + record.schema + "/" + record.tableName
       - 附加信息：databaseType, extractSql
     
    2. 若extractSql中涉及多表JOIN, 解析SQL提取所有源表
       → 可能生成多个EXT_DB源节点
     
    3. 构造目标节点（LAKE_TABLE）
    4. 根据dataType处理
    5. 构造血缘边（collect_method = "DB"）
    6. 校验 + 保存
```

**4.1.2.5 Kafka采集血缘生成逻辑**

```
输入：Kafka采集接口返回的data列表
处理流程：
  FOR EACH record IN data:
    1. 构造源节点（EXT_KAFKA）
       - node_unique_key = record.clusterServer + "/" + record.topicName
       - 若topicName含多个topic（逗号分割），拆分为多个源节点
     
    2. 构造目标节点（LAKE_TABLE）
    3. 根据dataType处理
    4. 构造血缘边（collect_method = "KAFKA"）
    5. 校验 + 保存
```

**4.1.2.6 表结构清单同步逻辑**

```
输入：表结构清单接口返回的data列表
处理流程：
  FOR EACH record IN data:
    1. 查找已有LAKE_TABLE节点（by tableName）
    2. 更新/新增节点元数据：
       - tableCnName, createSql, tableKey, partition
    3. 解析createSql提取字段信息：
       - 更新lineage_node_column表
    4. 根据dataType处理增删改
```

#### 4.1.3 数据同步调度设计

```yaml
Job: lineage_ingest_sync
Schedule: 0 0 1 * * ?  # 每日凌晨1:00
Steps:
  1. 调用表结构清单接口（先更新元数据）
     - 首次fullAmount=1, 后续fullAmount=0
  2. 调用文件采集方式接口
  3. 调用数据库采集方式接口
  4. 调用Kafka采集方式接口
  5. 血缘一致性校验
  6. 同步到Neo4j图数据库
  7. 生成同步报告（成功/失败/新增/更新/删除数量）
Error Handling:
  - 接口超时重试3次，间隔30s
  - 失败发送告警通知
  - 支持手动触发重新同步
```

### 4.2 统一血缘接入标准校验能力

#### 4.2.1 校验规则引擎设计

```java
public interface LineageNodeValidator {
    ValidationResult validate(LineageNode node);
}
```

| 节点类型 | 校验规则 | 说明 |
|---------|---------|------|
| EXT_DB（湖外业务系统） | ① systemName不为空 ② serverIP格式合法 ③ tableName符合命名规范 ④ 系统编码在已注册系统中存在 | 确保业务系统节点信息完整合规 |
| EXT_FILE（文件服务器） | ① protocol在支持列表内（sftp/ftp/hdfs） ② serverIP格式合法 ③ filePath不为空 ④ fileNameFormat中变量占位符合规 | 确保文件服务器节点可达 |
| LAKE_TABLE（湖内表） | ① clusterID在注册集群列表中 ② database存在 ③ tableName符合湖内命名规范（如前缀校验） ④ 表在元数据中心已注册 | 确保湖内表节点准确 |
| EXT_KAFKA（Kafka） | ① clusterServer格式合法 ② topicName不为空 | 确保Kafka节点合规 |
| OUT_ES（ES输出） | ① ES集群地址合法 ② indexName不为空 | 确保ES输出节点合规 |

#### 4.2.2 校验流程

```
血缘数据 → 节点类型识别 → 加载对应校验规则集 → 逐项校验 → 
  ├─ 全部通过 → 标记VALID → 进入保存流程
  └─ 存在校验失败 → 标记INVALID → 记录失败原因 → 通知管理员
```

### 4.3 统一血缘接入标准保存能力

#### 4.3.1 保存流程设计

```
校验通过的血缘数据 → 
  1. 节点去重判断（by node_unique_key）
     ├─ 已存在 → 更新节点属性
     └─ 不存在 → 新增节点
  2. 边去重判断（by source_node_id + target_node_id + lineage_stage）
     ├─ 已存在 → 更新边属性
     └─ 不存在 → 新增边
  3. MySQL事务提交
  4. 异步同步到Neo4j
  5. 记录操作日志
```

#### 4.3.2 幂等性设计

- 节点以`node_unique_key`为幂等键
- 边以`source_node_id + target_node_id + lineage_stage + collect_method`为幂等键
- 支持重复执行不产生重复数据

### 4.4 湖内数据共享开放血缘节点生成

#### 4.4.1 共享到行云文件服务器

```
触发：对接共享门户，共享操作完成后回调
生成血缘链路：LAKE_TABLE → OUT_FILE（行云）
  - 源节点：已有湖内表节点
  - 目标节点：新建OUT_FILE节点
    - node_unique_key = "XINGYUN://" + fileServerID + filePath
  - 边：lineage_stage = "OUTPUT", lineage_type = "TABLE"
```

#### 4.4.2 共享到非行云文件服务器

```
触发：对接共享门户，共享操作完成后回调
生成血缘链路：LAKE_TABLE → OUT_FILE_EXT
  - 目标节点：新建OUT_FILE_EXT节点
    - node_unique_key = protocol + "://" + serverIP + ":" + port + filePath
```

#### 4.4.3 共享到中心内系统数据库

```
触发：对接共享门户
生成血缘链路：LAKE_TABLE → OUT_DB
  - 目标节点：新建OUT_DB节点
    - node_unique_key = serverIP + ":" + port + "/" + instance + "/" + schema + "/" + tableName
```

#### 4.4.4 发布到ES提供API查询

```
触发：ES发布操作完成后
生成血缘链路：LAKE_TABLE → OUT_ES → OUT_API
  - OUT_ES节点：esCluster + "/" + indexName
```

```java
public class TDPClusterAdapter implements ClusterAdapter {
  
    @Override
    public List<ScriptInfo> fetchScripts(ClusterConfig config) {
        // 1. 通过TDP API获取调度任务列表
        List<TaskInfo> tasks = tdpClient.getTaskList(config);
      
        // 2. 遍历获取每个任务的脚本内容
        List<ScriptInfo> scripts = new ArrayList<>();
        for (TaskInfo task : tasks) {
            ScriptInfo script = new ScriptInfo();
            script.setPath(task.getScriptPath());
            script.setContent(tdpClient.getScriptContent(task));
            script.setTaskId(task.getId());
            script.setClusterId(config.getClusterId());
            scripts.add(script);
        }
        return scripts;
    }
  
    @Override
    public String resolveTableUniqueKey(String clusterId, String tableName) {
        // 解析表的完整路径，生成跨集群唯一标识
        String database = extractDatabase(tableName);
        String pureName = extractTableName(tableName);
        return clusterId + "::" + database + "::" + pureName;
    }
```

```java
public interface LogCollector {
    List<LogEntry> collect(LogCollectConfig config, Date targetDate);
}

// 文件系统日志采集器
public class FileLogCollector implements LogCollector {
    @Override
    public List<LogEntry> collect(LogCollectConfig config, Date targetDate) {
        // 1. 根据path pattern和日期定位日志文件
        List<File> logFiles = resolveLogFiles(config.getLogPathPattern(), targetDate);
      
        // 2. 逐文件读取并解析
        List<LogEntry> entries = new ArrayList<>();
        for (File logFile : logFiles) {
            try (BufferedReader reader = new BufferedReader(
                    new InputStreamReader(new FileInputStream(logFile), config.getLogFileEncoding()))) {
                String line;
                StringBuilder sqlBuffer = new StringBuilder();
                boolean inSqlBlock = false;
              
                while ((line = reader.readLine()) != null) {
                    // 噪声过滤
                    if (matchesFilterPattern(line, config.getFilterPatterns())) continue;
                  
                    // SQL起始标记检测
                    if (!inSqlBlock && matchesPattern(line, config.getSqlStartMarker())) {
                        inSqlBlock = true;
                        sqlBuffer = new StringBuilder();
                        sqlBuffer.append(extractSqlPart(line, config.getSqlStartMarker()));
                        continue;
                    }
                  
                    // SQL收集中
                    if (inSqlBlock) {
                        sqlBuffer.append(" ").append(line.trim());
                        // SQL结束标记检测
                        if (matchesPattern(line, config.getSqlEndMarker())) {
                            inSqlBlock = false;
                            LogEntry entry = new LogEntry();
                            entry.setSql(sqlBuffer.toString());
                            entry.setTaskId(extractTaskId(line, config.getTaskIdPattern()));
                            entry.setLogFile(logFile.getAbsolutePath());
                            entries.add(entry);
                        }
                    }
                }
            }
        }
        return entries;
    }
}

// TDP平台日志采集器
public class TDPLogCollector implements LogCollector {
    @Override
    public List<LogEntry> collect(LogCollectConfig config, Date targetDate) {
        // 1. 调用TDP API获取指定日期的任务执行记录
        List<TaskExecution> executions = tdpClient.getExecutions(targetDate);
      
        // 2. 逐任务获取执行日志
        List<LogEntry> entries = new ArrayList<>();
        for (TaskExecution exec : executions) {
            String logContent = tdpClient.getTaskLog(exec.getExecutionId());
            // 3. 从日志内容提取SQL
            List<String> sqls = extractSqlsFromLog(logContent);
            for (String sql : sqls) {
                LogEntry entry = new LogEntry();
                entry.setSql(sql);
                entry.setTaskId(exec.getTaskId());
                entries.add(entry);
            }
        }
        return entries;
    }
}
```

#### 10.5.1 合并策略

```java
public class LineageMergeStrategy {
  
    /**
     * 合并静态解析与日志解析的血缘结果
     */
    public MergeResult merge(List<LineageEdge> staticResult, List<LineageEdge> logResult) {
        MergeResult result = new MergeResult();
      
        // 1. 以日志解析为基准（变量已替换，更准确）
        Map<String, LineageEdge> logMap = logResult.stream()
            .collect(Collectors.toMap(e -> e.getSourceNodeId() + "|" + e.getTargetNodeId(), e -> e));
      
        Map<String, LineageEdge> staticMap = staticResult.stream()
            .collect(Collectors.toMap(e -> e.getSourceNodeId() + "|" + e.getTargetNodeId(), e -> e));
      
        // 2. 日志有、静态也有 → 采用日志结果（置信度更高）
        for (Map.Entry<String, LineageEdge> entry : logMap.entrySet()) {
            if (staticMap.containsKey(entry.getKey())) {
                LineageEdge merged = entry.getValue();
                merged.setConfidence(0.98);
                merged.setParseMethod("LOG+AST");
                result.addConfirmed(merged);
            } else {
                // 3. 日志有、静态无 → 新增（日志发现的增量血缘）
                LineageEdge added = entry.getValue();
                added.setConfidence(0.90);
                added.setParseMethod("LOG");
                result.addNewDiscovered(added);
            }
        }
      
        // 4. 静态有、日志无 → 保留但降低置信度（可能是条件分支未执行的路径）
        for (Map.Entry<String, LineageEdge> entry : staticMap.entrySet()) {
            if (!logMap.containsKey(entry.getKey())) {
                LineageEdge kept = entry.getValue();
                kept.setConfidence(0.70);
                kept.setParseMethod("AST_ONLY");
                result.addStaticOnly(kept);
            }
        }
      
        return result;
    }
}
```

#### 12.3.3 AI结果融合引擎

```java
@Service
public class AILineageFusionEngine {
  
    /**
     * 融合AST解析结果和AI补充结果
     */
    public FinalLineageResult fuse(ASTLineageResult astResult, AILineageResult aiResult) {
        FinalLineageResult finalResult = new FinalLineageResult();
      
        // === 表级血缘融合 ===
        // 1. AST基础结果（高置信度）
        for (TableLineage tl : astResult.getTableLineages()) {
            tl.setConfidence(0.98);
            tl.setParseMethod("AST");
            finalResult.addTableLineage(tl);
        }
      
        // 2. AI补充的表级血缘
        for (SupplementTableLineage stl : aiResult.getSupplementTableLineages()) {
            String key = stl.getSourceTable() + "|" + stl.getTargetTable();
            if (!finalResult.hasTableLineage(key)) {
                // 新增的表级血缘
                TableLineage tl = convertToTableLineage(stl);
                tl.setParseMethod("AI");
                finalResult.addTableLineage(tl);
            }
            // 已存在则忽略（AST优先）
        }
      
        // === 字段级血缘融合 ===
        // 3. AST基础结果
        for (ColumnLineage cl : astResult.getColumnLineages()) {
            cl.setConfidence(0.95);
            cl.setParseMethod("AST");
            finalResult.addColumnLineage(cl);
        }
      
        // 4. AI补充的字段级血缘
        for (SupplementColumnLineage scl : aiResult.getSupplementColumnLineages()) {
            String key = scl.getSourceTable() + "." + scl.getSourceColumn() + "|" 
                       + scl.getTargetTable() + "." + scl.getTargetColumn();
            if (!finalResult.hasColumnLineage(key)) {
                ColumnLineage cl = convertToColumnLineage(scl);
                cl.setParseMethod("AI");
                finalResult.addColumnLineage(cl);
            }
        }
      
        // === 修正处理 ===
        // 5. 处理AI的修正建议
        for (Correction correction : aiResult.getCorrections()) {
            // 修正需人工确认，标记为PENDING_REVIEW
            CorrectionRecord record = new CorrectionRecord();
            record.setOriginal(correction.getOriginalRelation());
            record.setCorrected(correction.getCorrectedRelation());
            record.setReason(correction.getReason());
            record.setStatus("PENDING_REVIEW");
            finalResult.addCorrectionRecord(record);
        }
      
        return finalResult;
    }
}
```

```java
@Service
public class AIResultCacheService {
  
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
  
    private static final long CACHE_TTL = 7 * 24 * 3600; // 7天
  
    /**
     * 同一SQL内容不重复调用AI
     */
    public AILineageResult getOrCompute(String sql, String capability, 
                                         Supplier<AILineageResult> aiCall) {
        String cacheKey = "AI_LINEAGE:" + capability + ":" + DigestUtils.md5Hex(sql);
      
        // 查缓存
        String cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return JSON.parseObject(cached, AILineageResult.class);
        }
      
        // 调用AI
        AILineageResult result = aiCall.get();
      
        // 写缓存
        redisTemplate.opsForValue().set(cacheKey, JSON.toJSONString(result), 
                                         CACHE_TTL, TimeUnit.SECONDS);
        return result;
    }
}
```

| 应用服务 | 多节点部署，Nginx负载均衡 |
| MySQL | 主从复制，读写分离 |
| Neo4j | 集群模式（3节点） |
| Redis | 哨兵模式 |
| 定时任务 | XXL-Job分布式调度，单点执行 |
| AI服务 | 降级策略（超时回退AST结果） |

### 14.3 数据一致性设计

| 场景 | 一致性策略 |
|------|-----------|
| MySQL与Neo4j同步 | 先写MySQL，异步写Neo4j；失败进入重试队列 |
| 血缘增量更新 | 乐观锁+版本号控制 |
| 汇聚接口数据同步 | 全量+增量结合（首次全量，后续增量） |
| AI结果入库 | 事务保证，AI结果与AST结果原子合并 |

### 14.4 安全设计

| 安全项 | 方案 |
|--------|------|
| 接口认证 | AES加密Token（复用汇聚系统认证方案） |
| 权限控制 | RBAC模型，血缘备注编辑需特定权限 |
| 数据脱敏 | AI调用前对SQL中的敏感字段值进行脱敏 |
| 操作审计 | 所有血缘修改操作记录审计日志 |
| 接口限流 | 每系统每分钟100次调用限制 |

### 附录C：脚本解析示例

**天源迪科典型Shell脚本结构：**

```bash
#!/bin/bash
# 天源迪科 - 收入明细加工脚本
# Usage: sh income_detail.sh 20260515

day_id=$1
month_id=${day_id:0:6}
db_name="dw"

# 调用子脚本（嵌套调用）
sh /analysis_cloud/common/init_partition.sh ${day_id} ${db_name}

# 主SQL执行
spark-sql -e "
INSERT OVERWRITE TABLE ${db_name}.dwm_income_detail PARTITION(day_id='${day_id}')
SELECT 
    a.user_id,
    a.product_id,
    b.product_name,
    a.amount,
    a.amount * b.tax_rate AS tax_amount,
    CASE WHEN a.channel = '1' THEN '线上' ELSE '线下' END AS channel_name
FROM ${db_name}.dwd_order_detail a
LEFT JOIN ${db_name}.dim_product b ON a.product_id = b.product_id
WHERE a.day_id = '${day_id}'
  AND b.is_valid = 1
"

# 汇总SQL执行
sql="
INSERT OVERWRITE TABLE ${db_name}.dwa_income_summary PARTITION(month_id='${month_id}')
SELECT 
    product_id,
    SUM(amount) AS total_amount,
    COUNT(DISTINCT user_id) AS user_cnt
FROM ${db_name}.dwm_income_detail
WHERE day_id LIKE '${month_id}%'
GROUP BY product_id
"
spark-sql -e "${sql}"

echo "Job completed successfully"
```
